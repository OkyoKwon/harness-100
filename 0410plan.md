# 나만의 Harness 만들기 — 상세 기획서

> 작성일: 2026-04-10
> 상태: Reviewed (v2)
> 최종 검수: 아키텍처 리뷰 + 프론트엔드 실현 가능성 리뷰 반영

## 1. 개요

사용자가 기존 100개의 하네스를 사용하는 것을 넘어, **에이전트와 스킬을 직접 구성하여 나만의 커스텀 하네스를 처음부터 만들 수 있는 기능**을 제공한다. DB 없이 **localStorage**에 저장하며, 각 단계마다 **가이드/도움말**을 제공하여 진입장벽을 낮춘다.

### 기존 기능과의 차이

| 기능 | 기존 Composer | 기존 Customizer | **새 Builder** |
|------|--------------|----------------|----------------|
| 목적 | 기존 하네스 조합 | 기존 하네스 필드 수정 | 처음부터 새로 생성 |
| 에이전트 | 선택한 하네스에서 자동 | 이름/역할/설명 편집 | 직접 추가/삭제/설정 |
| 스킬 | 자동 병합 | 수정 불가 | 직접 구성 |
| 저장 | URL 파라미터 (임시) | 세션 메모리 (휘발) | **localStorage (영구)** |
| 내보내기 | ZIP/로컬 셋업 | ZIP/로컬 셋업 | ZIP/로컬 셋업 + JSON |

## 3. 데이터 모델

### 3.1 CustomHarness 타입

> **[리뷰 반영]** 기존 `Skill` 타입을 직접 재사용하고, `CustomAgent`는 `Agent`를 확장한다.
> `CustomSkill`은 삭제 — `Skill` 인터페이스와 구조가 동일하므로 중복 타입을 만들지 않는다.

```typescript
// custom-harness-types.ts

import type { Agent, Category, Skill, ExecutionStep, SkillMode, ExtensionSkill } from "./types";

/** 기존 Agent에 enabled 필드를 추가한 확장 타입 */
interface CustomAgent extends Agent {
  readonly enabled: boolean;
}

interface CustomHarness {
  readonly id: string;              // nanoid (예: "custom_a1b2c3")
  readonly slug: string;            // 자동 생성 (name 기반 kebab-case)
  readonly name: string;
  readonly description: string;
  readonly category: Category;
  readonly agents: ReadonlyArray<CustomAgent>;
  readonly skill: Skill;            // 기존 Skill 타입 직접 재사용
  readonly frameworks: ReadonlyArray<string>;
  readonly createdAt: string;       // ISO 8601
  readonly updatedAt: string;       // ISO 8601
  readonly version: number;         // 스키마 버전 (마이그레이션용)
  readonly baseHarnessId?: number;  // 기존 하네스에서 복제한 경우 원본 ID
}
```

### 3.2 Harness 변환 규칙 (custom-harness-converter.ts)

> **[리뷰 반영]** `exportAsHarness` 변환 계약을 명시적으로 정의한다.

```typescript
function toHarness(custom: CustomHarness): Harness {
  // enabled: false인 에이전트는 제외
  const activeAgents = custom.agents.filter(a => a.enabled);

  return {
    id: 0,                                // 가상 하네스 ID (merge-harnesses 패턴과 동일)
    slug: custom.slug,
    name: custom.name,
    description: custom.description,
    category: custom.category,
    agents: activeAgents.map(({ enabled, ...agent }) => agent),  // enabled 필드 제거
    skill: {
      ...custom.skill,
      // 비활성 에이전트를 실행 순서에서 제거
      executionOrder: custom.skill.executionOrder
        .filter(step => activeAgents.some(a => a.id === step.agentId)),
    },
    frameworks: custom.frameworks,
    agentCount: activeAgents.length,       // 활성 에이전트 수
    popularityRank: 0,                     // 커스텀 하네스는 랭킹 없음
    rawFiles: undefined,                   // 생성 시 자동 마크다운 생성
  };
}

/** 기존 Harness를 CustomHarness로 복제 */
function fromHarness(source: Harness): CustomHarness {
  return {
    id: nanoid(),
    slug: source.slug + "-custom",
    name: source.name,
    description: source.description,
    category: source.category,
    agents: source.agents.map(agent => ({ ...agent, enabled: true })),
    skill: source.skill,
    frameworks: [...source.frameworks],
    createdAt: new Date().toISOString(),
    updatedAt: new Date().toISOString(),
    version: 1,
    baseHarnessId: source.id,
  };
}
```

### 3.3 localStorage 스키마

```typescript
// 키: "harness100_custom_harnesses"
// 값: JSON.stringify(CustomHarnessStore)

interface CustomHarnessStore {
  readonly version: 1;
  readonly harnesses: ReadonlyArray<CustomHarness>;
}
```

**용량 전략**:
- localStorage 제한: ~5MB (브라우저별 상이)
- 하네스 1개 평균: ~3–5KB (에이전트 5개 기준)
- 예상 저장 가능: ~500개 이상
- 저장 시 용량 체크 → 초과 시 경고 토스트

### 3.4 마이그레이션 전략

> **[리뷰 반영]** 기존 `validation.ts`의 `parseStoredFavorites` 패턴을 따라 안전한 파싱 수행.

```typescript
function parseCustomHarnessStore(raw: unknown): CustomHarnessStore {
  if (typeof raw !== "string") return { version: 1, harnesses: [] };

  try {
    const parsed = JSON.parse(raw);
    if (!parsed || typeof parsed !== "object") return { version: 1, harnesses: [] };
    if (!Array.isArray(parsed.harnesses)) return { version: 1, harnesses: [] };

    switch (parsed.version) {
      case 1:
        return parsed as CustomHarnessStore;
      // 향후 case 2: return migrateV1toV2(parsed);
      default:
        return { version: 1, harnesses: [] };
    }
  } catch {
    return { version: 1, harnesses: [] };
  }
}
```

**순환 의존성 감지 알고리즘**:

> **[리뷰 반영]** 구체적인 알고리즘 명시.

```typescript
// builder-validation.ts
// Kahn's algorithm 기반 위상 정렬로 순환 감지
function detectCycle(agents: ReadonlyArray<CustomAgent>): string[] | null {
  const inDegree = new Map<string, number>();
  const adj = new Map<string, string[]>();

  for (const agent of agents) {
    inDegree.set(agent.id, 0);
    adj.set(agent.id, []);
  }
  for (const agent of agents) {
    for (const dep of agent.dependencies) {
      adj.get(dep)?.push(agent.id);
      inDegree.set(agent.id, (inDegree.get(agent.id) ?? 0) + 1);
    }
  }

  const queue = [...inDegree.entries()]
    .filter(([, deg]) => deg === 0)
    .map(([id]) => id);
  const sorted: string[] = [];

  while (queue.length > 0) {
    const node = queue.shift()!;
    sorted.push(node);
    for (const neighbor of adj.get(node) ?? []) {
      const newDeg = (inDegree.get(neighbor) ?? 1) - 1;
      inDegree.set(neighbor, newDeg);
      if (newDeg === 0) queue.push(neighbor);
    }
  }

  if (sorted.length < agents.length) {
    // 순환에 포함된 노드 반환
    return agents
      .filter(a => !sorted.includes(a.id))
      .map(a => a.name);
  }
  return null; // 순환 없음
}
```

- **모드 팁**: "모드는 같은 하네스를 다양한 방식으로 실행하는 방법입니다. 예: '빠른 리뷰' 모드는 핵심 에이전트만, '전체 리뷰' 모드는 모든 에이전트를 실행"

**검증 체크리스트** (자동 수행):
- [ ] 에이전트가 1개 이상 존재 (활성 에이전트 기준)
- [ ] 모든 활성 에이전트에 이름과 역할이 설정됨
- [ ] 실행 순서에 순환 의존성 없음
- [ ] 트리거 조건이 1개 이상 등록됨
- [ ] 비활성화된 에이전트가 실행 순서에 포함되지 않음

**액션 버튼**:
1. **저장**: localStorage에 저장 → 토스트 "저장 완료"
2. **ZIP 다운로드**: `toHarness()` 변환 후 기존 `zip-builder` 재사용
3. **로컬 셋업**: `toHarness()` 변환 후 기존 `local-writer` 재사용 (File System Access API)
4. **JSON 내보내기**: `.harness.json` 파일 다운로드 (공유용)

- 클릭 시 `fromHarness()` 변환 → Builder Step 1으로 이동
- `baseHarnessId`에 원본 ID 기록
- 모든 필드가 복제된 상태에서 사용자가 자유롭게 수정 가능
- 복제 시 카탈로그 metadata만 사용하고, 전체 detail JSON은 확인 후 fetch (네트워크 최적화)

## 6. 가이드 시스템 설계

### 6.1 인라인 가이드

각 Step의 입력 필드 아래에 짧은 도움말을 표시한다.

> **[리뷰 반영]** 기존 `Input` 컴포넌트에 `helperText`와 `errorMessage` prop을 확장하여 구현.

```
에이전트 이름
┌──────────────────────────────────────┐
│  code-reviewer                       │
└──────────────────────────────────────┘
  ↳ 역할을 나타내는 영문 kebab-case로 작성하세요    ← helperText (aria-describedby)
```

- 접근성: `aria-describedby`로 입력 필드와 가이드 연결
- 항상 표시 (토글 아님) — 첫 사용자를 위해

### 6.2 Step별 상단 안내 배너

각 Step 상단에 해당 단계의 목적과 핵심 가이드를 배너로 표시한다.

```
┌─ 에이전트 설정 가이드 ──────────────────────────────────┐
│ 에이전트는 하네스의 핵심 구성원입니다.                       │
│ 각 에이전트에게 명확한 역할을 부여하고, 필요한 도구만         │
│ 선택하세요. 의존성을 설정하면 실행 순서가 자동으로 결정됩니다. │
│                                              [닫기]  │
└──────────────────────────────────────────────────────┘
```

- 닫기 가능 (localStorage에 닫힘 상태 기억: `harness100_builder_guides_dismissed`)
- 다시 보기: Step 타이틀 옆 "?" 아이콘
- `role="status"`, 닫기 버튼에 `aria-label="가이드 닫기"`

### 6.3 도구 선택 툴팁

도구(tool) 체크박스 옆에 도구의 상세 설명을 표시한다.

> **[리뷰 반영]** 현재 Tooltip 컴포넌트가 없으므로 신규 `Tooltip` UI 컴포넌트를 구현한다 (~80줄).
> MVP에서는 CSS `title` 속성으로 대체 가능하며, Phase 3에서 스타일드 Tooltip으로 업그레이드.

```
[v] Bash  ←  "셸 명령어를 실행합니다. npm install, git 등
              시스템 명령어가 필요한 에이전트에 부여하세요."
```

### 6.4 검증 피드백

입력값이 부적절할 때 실시간 인라인 에러를 표시한다.

```
에이전트 이름
┌──────────────────────────────────────┐
│                                      │  ← 빈 입력 (aria-invalid="true")
└──────────────────────────────────────┘

의존성
```

### 7.2 훅: `useCustomHarnesses`

> **[리뷰 반영]** `useFavorites`의 패턴을 따라 `useEffect`에서 읽기 + `storage` 이벤트 크로스탭 동기화.

```typescript
interface UseCustomHarnesses {
  readonly harnesses: ReadonlyArray<CustomHarness>;
  readonly isLoading: boolean;
  readonly save: (harness: CustomHarness) => boolean;  // false면 용량 초과
  readonly remove: (id: string) => void;
  readonly duplicate: (id: string) => CustomHarness;
  readonly getById: (id: string) => CustomHarness | undefined;
  readonly toHarness: (id: string) => Harness;         // 기존 Harness 타입으로 변환
}
```

**구현 패턴**:
```typescript
function useCustomHarnesses(): UseCustomHarnesses {
  const [harnesses, setHarnesses] = useState<ReadonlyArray<CustomHarness>>([]);
  const [isLoading, setIsLoading] = useState(true);

  // 1. 마운트 시 useEffect에서 읽기 (SSR/hydration 불일치 방지)
  useEffect(() => {
    const raw = localStorage.getItem(STORAGE_KEYS.customHarnesses);
    const store = parseCustomHarnessStore(raw);
    setHarnesses(store.harnesses);
    setIsLoading(false);
  }, []);

  // 2. 크로스탭 동기화 (useFavorites 패턴)
  useEffect(() => {
    function onStorage(e: StorageEvent) {
      if (e.key === STORAGE_KEYS.customHarnesses) {
        const store = parseCustomHarnessStore(e.newValue);
        setHarnesses(store.harnesses);
      }
    }
    window.addEventListener("storage", onStorage);
    return () => window.removeEventListener("storage", onStorage);
  }, []);

  // 3. 저장 시 try-catch로 QuotaExceededError 감지
  // ...
}
```

## 8. 공유 방식

> **[리뷰 반영]** Base64 URL 공유는 실용성 부족으로 삭제. JSON 파일 공유를 메인 방식으로 채택.

### 8.1 JSON 파일 내보내기/가져오기 (Primary)

- **내보내기**: `CustomHarness` 객체를 JSON 파일로 다운로드 (`{slug}.harness.json`)
- **가져오기**: Builder 페이지에서 파일 선택 또는 드래그앤드롭 → 파싱 → 검증 → "가져오기" 확인 모달
- 크기 제한 없으며 가장 안정적인 공유 방식

```typescript
// 내보내기 형식
{
  "$schema": "harness-100-custom-v1",
  "exportedAt": "2026-04-10T12:00:00Z",
  "harness": { /* CustomHarness 전체 */ }
}
```

### 8.2 클립보드 공유 (Secondary)

- JSON을 클립보드에 복사 → 다른 사용자가 Builder 페이지에서 "클립보드에서 가져오기"
- `navigator.clipboard.readText()` 사용
- 사이즈가 작은 하네스(에이전트 1~2개)에 적합

## 13. 엣지 케이스 & 제약사항

| 상황 | 대응 |
|------|------|
| localStorage 비활성화 (시크릿 모드) | 세션 중에만 동작, 저장 시 "저장할 수 없습니다" 토스트 |
| localStorage 용량 초과 | `QuotaExceededError` 감지 → "저장 공간이 부족합니다" 안내 |
| 에이전트 0개로 저장 시도 | 검증 에러: "에이전트를 1개 이상 추가하세요" |
| 순환 의존성 | Kahn's algorithm 실시간 감지 → 에러 표시, 저장 차단 |
| 매우 긴 이름/설명 | 입력 시 글자수 제한 + 카운터 표시 |
| 동일 이름 하네스 | 경고 표시 (차단하지 않음 — ID가 다르므로) |
| 브라우저 탭 간 동기화 | `storage` 이벤트 리스닝 (favorites와 동일 패턴) |
| 이전 버전 데이터 로드 | `version` 필드 + `parseCustomHarnessStore`로 안전한 마이그레이션 |
| JSON 가져오기 데이터 손상 | 스키마 검증 실패 시 "유효하지 않은 파일입니다" 안내 |
| 위저드 중간 이탈 | `sessionStorage` 자동 임시 저장, 재진입 시 복구 제안 |
| 비활성 에이전트 + 내보내기 | `toHarness()` 변환 시 비활성 에이전트 제외 (명시적 필터링) |
| 에이전트 삭제 시 의존성 참조 | 삭제 전 해당 에이전트를 참조하는 다른 에이전트 경고 → 자동 의존성 제거 |

## 14. 교차 검수 결과 요약

### 수행한 검수

1. **아키텍처 리뷰**: 타입 정합성, 기존 코드 패턴 준수, 변환 파이프라인, SSR/hydration 호환성
2. **프론트엔드 실현 가능성 리뷰**: 4-Step 위저드 상태 관리, DnD 라이브러리, 누락 UI 프리미티브, 반응형 디자인

### CRITICAL 수정 (3건)

| # | 이슈 | 수정 내용 |
|---|------|----------|
| 1 | `CustomHarness.id`가 `string`인데 기존 `Harness.id`는 `number` → 내보내기 파이프라인 불일치 | `toHarness()` 변환 시 `id: 0` 사용 (merge-harnesses 패턴). Favorites/Composer와는 별도 관리 |
| 2 | `buildZip`/`local-writer`가 `Harness` 타입만 수용하는데 `CustomHarness`에 `agentCount`, `popularityRank`, `rawFiles` 누락 | `toHarness()` 변환 함수에서 `agentCount = agents.length`, `popularityRank = 0`, `rawFiles = undefined` 명시 |
| 3 | `CustomSkill`이 기존 `Skill`과 구조 동일 → 중복 타입 | `CustomSkill` 삭제, 기존 `Skill` 직접 재사용 |

### HIGH 수정 (5건)

| # | 이슈 | 수정 내용 |
|---|------|----------|
| 4 | `useHarnessBuilder` 단일 훅이 5개 이상의 관심사를 관리 | 4개 전문 훅으로 분리 + 얇은 오케스트레이터 |
| 5 | Base64 URL 공유가 5KB+ 하네스에서 2000자 제한 초과하여 대부분 실패 | URL 공유 삭제, JSON 파일 + 클립보드 공유로 대체 |
| 6 | Builder 페이지의 `"use client"` 경계 전략 미정의 | `builder-client.tsx`를 단일 클라이언트 경계로 지정 (composer-client 패턴) |
| 7 | 기존 `useModifications`, `AgentSidebar`, `AgentEditForm` 재사용 기회 누락 | 재사용 맵 추가 (Section 10) |
| 8 | `CustomAgent.enabled` 내보내기 시 동작 미정의 | `toHarness()`에서 `enabled: false` 에이전트 제외 명시 |

### MEDIUM 수정 (7건)

| # | 이슈 | 수정 내용 |
|---|------|----------|
| 9 | `Agent` → `CustomAgent` 변환 시 `enabled: true` 기본값 미명시 | `fromHarness()`에서 명시적 변환 |
| 10 | `migrateStore`에서 `JSON.parse(raw as string)` unsafe 캐스팅 | `parseCustomHarnessStore`로 대체 (validation.ts 패턴) |
| 11 | `STORAGE_KEYS` 추가 시 `as const` 패턴 준수 필요 | 기존 패턴 유지로 수정 |
| 12 | SSR/hydration 불일치 고려 없음 | `useEffect`에서 초기화 + `isLoading` 패턴 명시 |
| 13 | 순환 의존성 감지 알고리즘 미명시 | Kahn's algorithm 코드 추가 |
| 14 | DnD 라이브러리 미지정 | `@dnd-kit/core` + `@dnd-kit/sortable` 지정 (Phase 2) |
| 15 | MobileMenu에 Builder 링크 추가 누락 | 데스크톱 + 모바일 모두 추가로 수정 |
